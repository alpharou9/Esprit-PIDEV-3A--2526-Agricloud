quality, stability, and presentation-readiness of the existing application.

The work focused on four areas:

==================================================

The implementation order followed the highest-impact and lowest-risk principle.

The steps were organized as follows:

1. Add a few useful unit tests for real business rules

==================================================
4. UNIT TESTS
==================================================

4.1 Objective

The objective of the unit tests was to verify small but important business rules
already present in the application.

Instead of writing many artificial tests, only a few useful tests were added.
This makes the work easier to justify in class and keeps maintenance simple.

4.2 Business rules covered

The following business rules were tested:

- a product price must be strictly greater than zero
- a product quantity cannot be negative
- an invalid order status transition must be rejected
- cancelling and reopening an order must keep inventory and cancellation data consistent

4.3 Test files added

The following test files were added:

- tests\Entity\ProductValidationTest.php
- tests\Service\OrderStatusServiceTest.php

4.4 Summary of the tests

ProductValidationTest verifies that:
- a product with price 0 is rejected by validation
- a product with a negative quantity is rejected by validation

OrderStatusServiceTest verifies that:
- an order cannot jump from pending directly to shipped
- when a confirmed order is cancelled, the product stock is restored
- when the cancelled order is reopened, the stock is corrected again
- cancellation metadata is cleared correctly when the order is reopened

4.5 Command used

vendor\bin\phpunit tests

4.6 Result

Observed result:
OK (26 tests, 50 assertions)

This confirms that the tested business rules are working correctly and that the
existing code remains stable after the follow-up changes.

5. STATIC ANALYSIS WITH PHPSTAN
==================================================

5.1 Objective

PHPStan was added to improve reliability and detect typing and logic issues early.
The analysis was configured at level 5, which is strict enough to be meaningful
for a university sprint report, while still realistic for an existing Symfony project.

5.2 Configuration

PHPStan was added as a development dependency and configured to analyze the src folder.

Main configuration choices:
- level 5
- target directory: src
- memory limit adjusted for analysis

5.3 Command used

vendor\bin\phpstan analyse --memory-limit=1G --error-format=table

5.4 Before state

Before correction, PHPStan reported:
- 50 errors

The most relevant issues included:
- missing or imprecise types
- controller logic relying on loosely typed authenticated users
- possible null access
- redundant conditions
- imprecise repository typing in fixtures and services
- weaker typing in some entity and service methods

5.5 Corrections applied

Only safe and useful corrections were made.
The goal was not to refactor the whole project, but to remove the issues that had
the best quality impact without risking regressions.

Main corrections:

- Added a typed base controller helper for authenticated users
  This avoids repeating weakly typed getUser() handling in controllers.

- Updated several controllers to use a properly typed authenticated user
  This reduced null/type ambiguity in real application flows.

- Improved typing in fixtures
  Repository results were clarified to avoid analysis errors and unclear assumptions.

- Simplified redundant conditions in services
  This made the code easier to understand and reduced noise in static analysis.

- Improved typing in the Order entity
  Some nullable or loosely defined values were made clearer where safe.

5.6 After state

After correction, PHPStan reported:
- 0 errors

This is a strong measurable improvement and can be used as a clear before/after
indicator in the sprint report.

6. DOCTRINE DOCTOR ANALYSIS
==================================================

6.1 Objective

Doctrine Doctor was added to analyze Doctrine ORM usage at runtime through the Symfony
profiler. Unlike static analysis, this tool helps inspect the real behavior of the
application during page execution.

The goal was to review the most important marketplace pages and identify practical,
low-risk optimizations.

6.2 Pages reviewed

The following pages were selected because they are core to the user journey:

- marketplace
- product details
- cart
- checkout
- orders

6.3 What was checked

The analysis focused on:

- N+1 query risks
- lazy-loading patterns
- slow or heavy queries
- relation integrity/configuration concerns
- profiler stability and configuration issues

6.4 Main findings

The most relevant findings were:

- the cart page had a realistic risk of extra lazy-loading for seller information
  because the Twig template accesses product and seller data

- Doctrine tooling visibility was limited by an overly restrictive schema filter
  which could hide useful tables from analysis

- local environment stability was affected by a cache/profiler collision between
  two local copies of the same project

6.5 Optimizations applied

Only the highest-impact and lowest-risk changes were made.

Optimization 1: eager loading in the cart repository

The cart repository was updated to eager-load the product seller together with the
cart items. This reduces the risk of lazy-loaded extra queries during template rendering.

Why it matters:
- better runtime behavior on the cart page
- lower risk of N+1 patterns
- no functional change in business logic

Optimization 2: Doctrine schema filter cleanup

The Doctrine schema filter was broadened so the relevant project tables are included
correctly for development analysis tools.

Why it matters:
- cleaner analysis tooling
- fewer hidden tables in diagnostics
- more reliable development observations

Optimization 3: Symfony cache isolation per local project copy

Both local copies of the project were using the same temp cache folder.
This created collisions in:
- compiled containers
- profiler files
- development cache artifacts

A project-specific temp cache and log path was introduced in Kernel.php.

Why it matters:
- prevents collisions between local copies
- fixes unstable profiler behavior
- avoids misleading runtime class-loading errors
- improves development reliability on Windows/OneDrive setups

6.6 Result

After these changes:
- the profiler environment became more stable
- Doctrine Doctor configuration loaded correctly
- the cart page query pattern became safer
- runtime debugging became more trustworthy
